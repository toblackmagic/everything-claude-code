# Everything Claude Code - 使用说明

## 项目简介

**Everything Claude Code** 是一套由 Anthropic x Forum Ventures 黑客松获奖者打造的 Claude Code 插件集合。这套配置经过 10+ 个月的真实产品开发实战验证，提供生产就绪的代理（Agents）、技能（Skills）、钩子（Hooks）、命令（Commands）、规则（Rules）和 MCP 配置。

## 目录结构

```
everything-claude-code/
├── .claude-plugin/          # 插件和市场清单
├── agents/                  # 专业化子代理
├── skills/                  # 工作流定义和领域知识
├── commands/                # 斜杠命令
├── rules/                   # 永久遵循的指导原则
├── hooks/                   # 基于事件的自动化
├── scripts/                 # 跨平台 Node.js 脚本
├── contexts/                # 动态系统提示注入
├── mcp-configs/             # MCP 服务器配置
└── examples/                # 示例配置
```

## 安装方法

### 方式一：插件安装（推荐）

最简单的安装方式，直接将整个仓库安装为 Claude Code 插件：

```bash
# 添加此仓库为市场源
/plugin marketplace add affaan-m/everything-claude-code

# 安装插件
/plugin install everything-claude-code@everything-claude-code
```

或直接编辑 `~/.claude/settings.json`：

```json
{
  "extraKnownMarketplaces": {
    "everything-claude-code": {
      "source": {
        "source": "github",
        "repo": "affaan-m/everything-claude-code"
      }
    }
  },
  "enabledPlugins": {
    "everything-claude-code@everything-claude-code": true
  }
}
```

### 方式二：手动安装

如果需要更精细的控制，可以手动复制各组件：

```bash
# 克隆仓库
git clone https://github.com/affaan-m/everything-claude-code.git

# 复制代理
cp everything-claude-code/agents/*.md ~/.claude/agents/

# 复制规则
cp everything-claude-code/rules/*.md ~/.claude/rules/

# 复制命令
cp everything-claude-code/commands/*.md ~/.claude/commands/

# 复制技能
cp -r everything-claude-code/skills/* ~/.claude/skills/
```

### 设置（Setup）

#### 配置加载机制

Claude Code 按特定优先级顺序从多个来源加载配置：

```
优先级（从高到低）：
1. 环境变量
2. 项目配置 (.claude/)
3. 用户配置 (~/.claude/)
4. 插件配置
```

#### 配置位置

| 配置类型 | 位置 | 用途 |
|---------|------|------|
| 项目配置 | `项目根目录/.claude/` | 项目特定规则和技能 |
| 用户配置 | `~/.claude/` | 全局代理、命令、规则 |
| MCP 配置 | `~/.claude.json` | MCP 服务器连接 |

#### 工具系统

Claude Code 提供六个核心工具用于代码库交互：

| 工具 | 用途 | Token 消耗 | 典型用法 |
|-----|------|-----------|---------|
| `Read` | 读取文件内容 | 高（文件大小） | 检查代码、理解结构 |
| `Write` | 创建新文件 | 高（内容大小） | 生成新组件、测试 |
| `Edit` | 修改现有文件 | 中（差�大小） | 修复 Bug、重构 |
| `Grep` | 搜索文本模式 | 低-中 | 查找用法、定位代码 |
| `Glob` | 列出匹配文件 | 低 | 发现文件、构建列表 |
| `Bash` | 执行 shell 命令 | 低-高 | 运行测试、构建 |

**工具选择指南：**

```
查找文件 → 优先用 Glob（而非 Bash ls）
搜索文本 → 优先用 Grep（而非 Bash grep）
读取文件 → 使用 Read 工具
创建文件 → 使用 Write 工具
修改文件 → 优先用 Edit（而非 Write）
运行命令 → 使用 Bash 工具
```

### 哲学（Philosophy）

#### 核心设计原则

**1. 不可变性优先（Immutability First）**

```javascript
// ❌ 错误：直接修改
function updateUser(user, name) {
  user.name = name  // 变异！
  return user
}

// ✅ 正确：创建新对象
function updateUser(user, name) {
  return { ...user, name }
}
```

**2. 小文件优于大文件**

```
推荐结构：
├── components/
│   ├── Button.tsx          # 200 行
│   ├── Button.types.ts     # 50 行
│   ├── Button.utils.ts     # 80 行
│   └── Button.test.tsx     # 150 行

而非：
├── components/
    └── Button.tsx          # 800+ 行 ❌
```

**文件大小指南：**
- 典型：200-400 行
- 最大：800 行
- 超过 800 行 → 拆分

**3. 测试驱动开发（TDD）**

```
RED → GREEN → REFACTOR

1. RED:     编写失败的测试
2. GREEN:   编写最小代码使测试通过
3. REFACTOR: 改进代码，保持测试通过
```

**4. 并行优于串行**

```markdown
# ✅ 好：并行执行
启动 3 个代理并行执行：
1. 代理 1：auth.ts 安全分析
2. 代理 2：缓存系统性能审查
3. 代理 3：utils.ts 类型检查

# ❌ 差：不必要的串行
先执行代理 1，然后代理 2，然后代理 3
```

#### 代理委托哲学

**何时使用代理：**

| 场景 | 使用代理 | 理由 |
|-----|---------|------|
| 复杂功能请求 | planner | 需要详细规划 |
| 代码刚编写/修改 | code-reviewer | 质量保证 |
| Bug 修复或新功能 | tdd-guide | 测试优先 |
| 架构决策 | architect | 设计评估 |

**多视角分析：**

对于复杂问题，使用分角色子代理：
- 事实审查员
- 高级工程师
- 安全专家
- 一致性审查员
- 冗余检查员

#### 上下文管理哲学

**上下文 = 有限资源**

```
200k token 上下文窗口分配：
├── 系统提示：~20k tokens
├── 工具定义：~30k tokens（10 个 MCP × 3k）
├── 代码文件：~100k tokens
├── 对话历史：~40k tokens
└── 安全余量：~10k tokens
```

**优化策略：**

1. **Grep 先于 Read** - 定位相关文件后再读取
2. **Edit 优于 Write** - 更小的 diff = 更少上下文消耗
3. **限制 Bash 输出** - 管道到 head 或 grep
4. **使用 Glob 发现** - 比读取目录列表更便宜

#### 质量保证哲学

**多层防护体系：**

```
第 1 层：规则（Rules）
         ↓ 始终遵循的原则
第 2 层：技能（Skills）
         ↓ 工作流和最佳实践
第 3 层：钩子（Hooks）
         ↓ 自动化验证
第 4 层：代理（Agents）
         ↓ 专家级审查
```

**代码质量清单：**

完成工作前检查：
- [ ] 代码可读且命名良好
- [ ] 函数小（<50 行）
- [ ] 文件聚焦（<800 行）
- [ ] 无深度嵌套（>4 层）
- [ ] 适当的错误处理
- [ ] 无 console.log 语句
- [ ] 无硬编码值
- [ ] 使用不可变模式
- [ ] 80%+ 测试覆盖率

## 斜杠命令详解

### /tdd - 测试驱动开发

**描述：** 强制执行 TDD 工作流，先定义接口，先生成测试，然后实现最小代码。

**使用场景：**
- 实现新功能时
- 添加新函数/组件时
- 修复 Bug 时（先写重现 Bug 的测试）
- 重构现有代码时

**TDD 循环：**

```
RED（红）→ GREEN（绿）→ REFACTOR（重构）→ 重复

RED:      编写失败的测试
GREEN:    编写最小代码使测试通过
REFACTOR: 改进代码，保持测试通过
REPEAT:   下一个功能/场景
```

**使用示例：**


用户: /tdd 我需要一个计算市场流动性评分的函数
代理 (tdd-guide):
TDD 会话：市场流动性评分计算器

1. 步骤 1：定义接口（脚手架）

```typescript
// lib/liquidity.ts
export interface MarketData {
  totalVolume: number
  bidAskSpread: number
  activeTraders: number
  lastTradeTime: Date
}

export function calculateLiquidityScore(market: MarketData): number {
  throw new Error('Not implemented')
}
```

2. 步骤 2：编写失败测试（RED）

```typescript
// lib/liquidity.test.ts
import { calculateLiquidityScore } from './liquidity'

describe('calculateLiquidityScore', () => {
  it('should return high score for liquid market', () => {
    const market = {
      totalVolume: 100000,
      bidAskSpread: 0.01,
      activeTraders: 500,
      lastTradeTime: new Date()
    }
    const score = calculateLiquidityScore(market)
    expect(score).toBeGreaterThan(80)
  })
})
```

3. 步骤 3：运行测试 - 验证失败

4. 步骤 4：实现最小代码（GREEN）

5. 步骤 5：运行测试 - 验证通过

6. 步骤 6：重构（IMPROVE）

7. 步骤 7：验证覆盖率（80%+）

**最佳实践：**
- ✅ 先写测试，后写实现
- ✅ 实现前验证测试失败
- ✅ 编写最小代码使测试通过
- ✅ 测试通过后重构
- ✅ 目标 80%+ 覆盖率

**相关命令：**
- 先用 `/plan` 理解要构建什么
- 用 `/tdd` 实现并测试
- 用 `/code-review`（通用）、`/frontend-review`（React）、`/rn-review`（React Native）审查实现

### /frontend-review - 前端代码审查

**描述：** React 和 React Native 共享代码审查，涵盖组件、hooks、TypeScript、性能和安全。

**审查内容包括：**

- Hooks Best Practices（Hooks 规则、依赖数组、自定义 hook 模式、useEffect 清理）

- Performance (Shared)（React.memo 使用、useCallback/useMemo 模式、不必要的重渲染）
- Security (Shared)（输入验证、XSS 防止、HTTPS 强制）
- Testing Patterns（Testing Library 使用、测试 fixtures、mock 模式）

**使用场景：**
- React Web 项目代码审查
- React Native 共享代码审查（hooks/utils/types）
- Monorepo 中 Web + Mobile 共享代码

**与 /rn-review 的区别：**
- `/frontend-review` - 通用 React 模式（Web + Mobile 共享）
- `/rn-review` - React Native 特有问题（需要配合 `/frontend-review` 使用）

### /rn-review - React Native 代码审查

**描述：** React Native 专用代码审查，执行两层完整审查。

**两层审查架构：**

**Layer 1: Frontend Review（frontend-reviewer）**
- React Component Patterns（Props 接口、组件结构）
- Hooks Best Practices（Hooks 规则、依赖数组）
- TypeScript for React（类型定义、泛型组件）
- Performance (Shared)（React.memo、useCallback/useMemo）
- Security (Shared)（输入验证、XSS、HTTPS）

**Layer 2: RN-Specific Review（rn-reviewer）**
- Platform Compatibility（Platform API、Safe Area、.ios./.android. 文件）
- FlatList Optimization（keyExtractor、removeClippedSubviews、maxToRenderPerBatch）
- Native Integration（StyleSheet vs inline、FastImage、useNativeDriver、Bridge 调用）
- RN Security（AsyncStorage vs Keychain、Deep link 验证、Native 模块权限）
- RN Memory Management（AppState 清理、NetInfo 取消订阅、Native 计时器）

**审查类别：**

**CRITICAL（必须修复）：**
- FlatList 缺少 keyExtractor（导致崩溃和性能问题）
- AsyncStorage 存储敏感数据（令牌、密码）
- 缺少 HTTPS API 调用
- Deep link 注入漏洞
- 内存泄漏（事件监听器、计时器、订阅未清理）

**HIGH（应该修复）：**
- 缺少组件 memoization（性能）
- 缺少 useCallback/useMemo
- Platform 特定代码未正确组织
- 缺少导航 prop 类型
- 缺少安全区域处理

**使用场景：**
- React Native 项目完整代码审查
- 移动应用性能优化
- iOS/Android 平台兼容性检查

**示例工作流：**

```text
用户: /rn-review

代理 (frontend-reviewer):
# Frontend Code Review Report
[审查共享 React 模式...]

代理 (rn-reviewer):
# React Native Code Review Report
[审查 RN 特有问题...]
```

### /setup-pm - 配置包管理器

**描述：** 配置首选包管理器（npm/pnpm/yarn/bun）。

**检测优先级：**

1. 环境变量：`CLAUDE_PACKAGE_MANAGER`
2. 项目配置：`.claude/package-manager.json`
3. package.json：`packageManager` 字段
4. 锁文件检测
5. 全局配置：`~/.claude/package-manager.json`
6. 回退：首个可用包管理器

**使用方式：**

```bash
# 检测当前设置
node scripts/setup-package-manager.js --detect

# 设置全局偏好
node scripts/setup-package-manager.js --global pnpm

# 设置项目偏好
node scripts/setup-package-manager.js --project bun

# 列出可用的包管理器
node scripts/setup-package-manager.js --list
```

**环境变量设置：**

```bash
# macOS/Linux
export CLAUDE_PACKAGE_MANAGER=pnpm

# Windows (PowerShell)
$env:CLAUDE_PACKAGE_MANAGER = "pnpm"
```

### 配置方法

将 `mcp-configs/mcp-servers.json` 中所需的服务器复制到 `~/.claude.json`。

**重要：** 将 `YOUR_*_HERE` 占位符替换为实际 API 密钥。

### 上下文窗口管理

**重要：** 不要同时启用所有 MCP。启用太多工具会使 200k 上下文窗口缩减到 70k。

**经验法则：**
- 配置 20-30 个 MCP
- 每个项目保持 10 个以下启用
- 保持 80 个以下工具活动

使用项目配置中的 `disabledMcpServers` 禁用未使用的服务器。

## tmux 工作流

### 为什么使用 tmux

长时间运行的命令（开发服务器、测试、构建）应该在 tmux 会话中运行：
- 会话持久化（断开 SSH 不会终止）
- 易于日志访问
- 多窗口并行工作

### tmux 基本命令

```bash
# 创建新会话
tmux new -s dev

# 分离会话（保持运行）
Ctrl+B, then D

# 重新连接
tmux attach -t dev

# 列出会话
tmux ls

# 终止会话
tmux kill-session -t dev
```

### 推荐工作流

```bash
# 1. 创建 tmux 会话
tmux new -s dev

# 2. 在 tmux 中启动开发服务器
npm run dev

# 3. 分离会话（Ctrl+B, D）
# 服务器继续在后台运行

# 4. 随时重新连接查看日志
tmux attach -t dev
```

## 常见问题解答

### Q: 为什么我的开发服务器被阻止？

A: 此插件默认阻止在 tmux 外运行开发服务器。使用：

```bash
tmux new -s dev
npm run dev
# 然后 Ctrl+B, D 分离
```

### Q: 如何禁用特定的 MCP 服务器？

A: 在项目 `.claude.json` 中添加：

```json
{
  "disabledMcpServers": [
    "github",
    "clickhouse"
  ]
}
```

### Q: 80% 覆盖率要求是否灵活？

A: 对于核心业务逻辑，目标是 100%。对于样板代码或简单的 UI 组件，可以接受较低的覆盖率，但需说明原因。

### Q: 何时应该使用代理而不是直接让 Claude 做？

A:
- **使用代理：** 复杂任务、需要专业知识、需要一致性
- **直接请求：** 简单任务、快速迭代、探索性工作
